# Web3Auth PnP Modal SDK v10 Migration Guide (Unified)

This guide will help you upgrade your Web3Auth PnP SDK integration from v9 (whether you were using
the Modal or the `@web3auth/no-modal` SDK) to the **unified Web3Auth PnP Modal SDK v10**.

Version 10 significantly simplifies your Web3Auth integration by consolidating into a single
powerful SDK. It centralizes configuration in the Web3Auth Developer Dashboard and removes the
complexity of managing separate adapters, connections (previously verifiers), and manual blockchain
configuration from your frontend code. React integrations now exclusively use a hooks-based
approach. Direct connections (for developers implementing a custom UI, a capability previously
prominent in the No-Modal SDK) are now seamlessly available within this unified Modal SDK.

:::tip Who is this guide for?

This guide is for developers migrating from:

- `@web3auth/modal` v9.x
- `@web3auth/no-modal` v9.x

Both groups will migrate to the new, unified `@web3auth/modal` v10 SDK.

:::

:::note Before You Start

To ensure a smooth migration, please consider the following steps:

- **Backup Your Project:** It's crucial to have a backup or a version control checkpoint of your
  project before starting the migration process.
- **Use Latest v9.x Version:** Ensure your application is on the latest Web3Auth v9.x version of
  your respective SDK (`@web3auth/modal` or `@web3auth/no-modal`). This can provide helpful
  deprecation warnings.
- **Web3Auth Developer Dashboard Access:** A key aspect of v10 is that many configurations (like
  verifiers, chain settings, Smart Accounts, and modal UI options) are now managed via the
  [Web3Auth Developer Dashboard](https://dashboard.web3auth.io/). Make sure you have access to your
  project settings on the dashboard.
- **Understand the Scope:** Briefly review the "Why these changes?" and "Migration Overview"
  sections to understand the scope and nature of the upgrade.
- **Allocate Sufficient Time:** Go through this guide once entirely to estimate the time and effort
  required for your specific integration.

:::

:::tip Key Change in v10: Dashboard-Centric Configuration & Unified SDK

Remember, a fundamental shift in v10 is moving many configurations that were previously in your
client-side code (like connection details, chain configurations, Smart Account settings, and even
modal UI customizations) to the **Web3Auth Developer Dashboard**. The second major shift is the
consolidation into a **single `@web3auth/modal` SDK**, which now supports both modal-based and
direct connection flows (for custom UI implementations).

:::

## Why these changes?

Web3Auth v9 provided developers with distinct SDKs for modal and non-modal integrations, offering
targeted solutions and flexibility. As the Web3 landscape evolves and to further enhance the
developer experience, Web3Auth v10 introduces significant refinements aimed at simplifying common
integration patterns, reducing boilerplate, and centralizing configuration.

Building on the foundation of v9, key areas of evolution in v10 include:

- **A Unified & Versatile SDK (`@web3auth/modal`):** To offer greater versatility and simplify the
  initial SDK choice, v10 consolidates all functionalities into a single `@web3auth/modal` SDK. This
  powerful SDK allows developers to seamlessly implement pre-built modal UIs or create custom user
  interfaces with direct connections to authentication providers, all using one consistent API.
- **Dashboard-Centric Configuration:** Many configurations, such as login methods, connection
  details (formerly verifiers), Smart Account settings, and chain specifics, are now managed through
  the Web3Auth Developer Dashboard. This shift centralizes control, reduces client-side code, and
  makes updates across projects more straightforward.
- **Streamlined Login & Connection Management:** V10 moves towards a more declarative approach.
  Manual client-side adapter registration and configuration are minimized, with login methods
  primarily defined on the dashboard and client-side connection calls becoming more direct and
  intuitive.
- **Automated Blockchain Setup:** For standard blockchains, v10 reduces the need for manual
  `chainConfig` and `privateKeyProvider` setup in the client by handling these configurations
  automatically based on your dashboard settings.
- **Integrated Smart Account Functionality:** Smart Account integration is now more cohesive. The
  primary configuration is managed on the dashboard, and the functionality is built directly into
  the main SDK, removing the need for a separate Smart Account provider package for most common use
  cases.
- **Simplified Multi-Factor Authentication (MFA):** MFA setup has been streamlined, offering more
  direct configuration options within the SDK initialization or per connection.
- **Modernized React Experience:** For React applications, v10 fully embraces a hooks-based
  architecture, with the `Web3AuthProvider` automatically handling SDK initialization for a cleaner
  development workflow.

The result is a cleaner, more declarative, and more maintainable integration experience—especially
for teams maintaining apps across multiple auth flows and chains, all using one SDK.

## Migration Overview

You'll migrate your app following these steps. Note that some steps might be more relevant if you
are coming from the v9 No-Modal SDK versus the v9 Modal SDK.

1.  **Install the Unified `@web3auth/modal` v10 SDK** and remove old packages.
2.  **Update SDK Initialization**: Adjust how `Web3Auth` or `Web3AuthProvider` is initialized.
3.  **Blockchain & Chain Configurations**: Transition to dashboard-managed chain settings.
4.  **Remove Adapters & Update Login/Connection Logic**: Shift from client-side adapter
    configuration to dashboard settings and streamlined `connect`/`connectTo` calls.
5.  **Update Verifier Usage**: Replace client-side `verifier` and `verifierSubIdentifier` with
    `authConnectionId` configured on the dashboard.
6.  **Migrate React Usage to Hooks (React only)**: This is mandatory for React users.
7.  **Update Multi-Factor Authentication (MFA) Setup**: Adapt to the new MFA configuration methods.
8.  **Update Smart Accounts Setup (optional)**.
9.  **Review Deprecated APIs**.
10. **Confirm Changes** with a migration summary.

## Installation

Install the latest v10 unified SDK package:

```bash
npm install @web3auth/modal@latest
```

Remove deprecated and now-consolidated packages if present:

- `@web3auth/no-modal` (if migrating from v9 No-Modal)
- `@web3auth/base`
- `@web3auth/auth-adapter` (and other specific adapter packages)
- `@web3auth/account-abstraction-provider`
- `@web3auth/ethereum-provider` (and other specific chain providers if you were using them directly
  with Web3Auth initialization for standard EVM chains)

:::warning Important

Ensure these packages are fully removed to avoid unexpected behavior and to leverage the unified v10
architecture.

:::

renamed to `useSFAKey` (v10); this is a breaking change if you were using it to retrieve

Key changes:

- Import `Web3Auth` from `@web3auth/modal`.
- `modalConfig` is now passed directly into the `Web3Auth` constructor. It's used to customize the

  or dashboard-configured social/custom connections (via `WALLET_CONNECTORS.AUTH`) are displayed and
  their order.
- Explicit `privateKeyProvider` and `chainConfig` for standard EVM chains are removed from
  constructor (managed by dashboard).
- `init()` is the sole initialization method for the class-based SDK. `initModal()` is deprecated.
- The `useCoreKitKey` parameter (boolean) is now `useSFAKey`.

Key changes for React:

- Use `Web3AuthProvider` from `@web3auth/modal/react`.
- The `Web3AuthProvider` handles SDK initialization automatically; you do not need to call `init()`

- Pass `initParams` (with `clientId`, `web3AuthNetwork`, optional `modalConfig`, `mfaSettings`,

- The `useCoreKitKey` parameter (boolean) in `initParams` is now `useSFAKey`.
- Explicit `privateKeyProvider` is no longer passed for standard EVM chains.

## 2. Blockchain & Chain Configurations: Embrace the Dashboard

This is a major simplification. You no longer need to manually set up `chainConfig` objects (for
standard chains like Ethereum, Polygon, BNB, etc.) or `privateKeyProvider` instances in your
client-side code when initializing Web3Auth.

**What to do:**

1.  **Remove Client-Side Code:** Delete any code where you were creating
    `EthereumPrivateKeyProvider`, `SolanaPrivateKeyProvider`, or other chain-specific providers
    solely for Web3Auth initialization.
2.  **Remove `chainConfig` from Constructor/Options:** Delete `chainConfig` passed to the `Web3Auth`
    constructor or `Web3AuthProvider` options if it was for a standard chain now configurable on the
    dashboard.
    - _Exception for React_: You still pass a primary `chainConfig` to `Web3AuthProvider` to set the
      initial chain context for the hooks and provider instance it exposes.
3.  **Configure on Web3Auth Dashboard:**
    - Log in to the [Web3Auth Developer Dashboard](https://dashboard.web3auth.io/).
    - Navigate to your Project > "Chains".
    - Here, you can enable/disable standard EVM and Solana chains. The SDK will automatically use
      the correct configurations for these.
    - You can also add **Custom Chains** (EVM or other types if supported) by providing their RPC
      endpoint, chain ID, currency symbol, etc.
    - ![Chains on Dashboard](https://i.ibb.co/4nCD2GTJ/chains.gif)
    - **Infura RPC for Standard Chains:** Notice that many standard EVM chains on the dashboard come
      with an Infura icon. This indicates that Web3Auth bundles premium Infura RPC access for these
      chains, ensuring reliable connectivity without you needing to manage your own RPC keys for
      them within Web3Auth.

## 3. Remove Adapters & Update Login/Connection Logic

Client-side adapter configuration (e.g., `AuthAdapter`, `OpenLoginAdapter`) is deprecated. Login
methods and their specific configurations are now primarily managed on the Web3Auth Developer
Dashboard under your Project > "Authentication" section, which includes "Social Connections" and
"Custom Connections".

**Modal Flows:**

If you are using the Web3Auth Modal UI:

- **v9:** You might have configured adapters and passed them to `web3auth.configureAdapter()` or
  initialized `Web3Auth` with an `OpenLoginAdapter` instance in `modalConfig`.
- **v10:**
  1.  Remove all code related to `web3auth.configureAdapter()`.
  2.  Remove explicit adapter instantiation for `modalConfig`.
  3.  Login methods displayed in the modal are now primarily controlled by your **Dashboard
      settings** under Project > "Authentication".
      - ![Connections on Dashboard](https://i.ibb.co/nNG9Snrq/auth-section.gif)
  4.  You can still customize which external wallet connectors appear and their order (e.g.
      WalletConnect, MetaMask), or influence the display of dashboard-configured social/custom
      connections (those under `WALLET_CONNECTORS.AUTH`), using `modalConfig` in the `Web3Auth`
      constructor or `Web3AuthProvider` options. However, the core connection details (like
      `authConnectionId` for custom auth) are referenced from dashboard configurations.

**Direct Connection Flows (for custom UI - capability previously in No-Modal SDK, now in unified
Modal SDK):**

- **v9 (`@web3auth/no-modal`):** You used
  `web3auth.connectTo(WALLET_ADAPTERS.OPENLOGIN, { loginProvider, ... })` or
  `web3auth.connectTo(WALLET_ADAPTERS.AUTH, { ... })` with detailed configurations for
  `loginProvider`, `verifier`, `clientId` (for social), `idToken` (for JWTs), etc.
- **v10 (`@web3auth/modal`):**
  1.  **Non-React:** Use the `web3auth.connectTo(connectorName, options)` method. The
      `connectorName` comes from `WALLET_CONNECTORS` (e.g., `WALLET_CONNECTORS.AUTH` for custom
      authentication connections configured on the dashboard, or specific external wallet connectors
      like `WALLET_CONNECTORS.METAMASK`). The `options` object for `WALLET_CONNECTORS.AUTH`
      primarily takes an `authParams` field, which includes:
      - `connection`: From `AUTH_CONNECTION` enum (e.g., `AUTH_CONNECTION.GOOGLE`,
        `AUTH_CONNECTION.CUSTOM_JWT`). This specifies the type of the dashboard-configured
        connection.
      - `authConnectionId`: Your `authConnectionId` (the unique ID of your Connection from the
        Web3Auth Dashboard).
      - `idToken` (if `authParams.connection` is `AUTH_CONNECTION.CUSTOM_JWT`).
      - `mfaLevel` (optional, see MFA section).
      - Other specific parameters as needed by the connection type.
  2.  **React:** Use the `connectTo(connectorName, options)` function from the `useWeb3AuthConnect`
      hook for direct connections. The parameters are similar to the non-React `connectTo`. If you
      want to display the pre-built Web3Auth Modal, you would use the `connect()` function
      (typically without arguments) from the same hook.

**Example: Direct Connection with Custom JWT (Firebase Example - React Hooks)**

This pattern, for building a custom UI with direct connections, was common with the v9 No-Modal SDK
and is now achieved with the unified `@web3auth/modal/react` hooks using `connectTo`.

_v10 Unified Modal (React Hooks)_:

```tsx
// Adapted from your Firebase JWT example
import { useWeb3AuthConnect } from "@web3auth/modal/react";
import { WALLET_CONNECTORS, AUTH_CONNECTION } from "@web3auth/modal";
// ... Firebase setup ...

const { connectTo, loading, error } = useWeb3AuthConnect(); // Use connectTo for direct connections

const loginWithFirebaseGithub = async () => {
  try {
    // ... (Firebase sign-in logic to get idToken) ...
    const idToken = await result.user.getIdToken(true);

    await connectTo({
      walletConnector: WALLET_CONNECTORS.AUTH,
      authParams: {
        connection: AUTH_CONNECTION.CUSTOM_JWT,
        authConnectionId: "w3a-firebase-demo", // Your Auth Connection ID from Dashboard
        idToken, // The JWT from Firebase
        // extraLoginOptions: { isUserIdCaseSensitive: false } // Still available if needed for specific connection settings
      },
    });
  } catch (err) {
    console.error("Login error:", err);
  }
};
```

## 4. Update Connection (Verifier) Usage

Client-side declaration of `verifier` (often the verifier name in v9) and `verifierSubIdentifier`
(for social logins within an aggregate verifier in v9) is deprecated. In v10, these are replaced by
`authConnectionId` and related configurations on the dashboard.

**What to do:**

1.  **Configure Connections on Dashboard:** All connection configurations, including setting up JWT
    connections (specifying JWKS URL, audience, user claim/`verifierIdField`) and linking social
    logins, are now done on the [Web3Auth Developer Dashboard](https://dashboard.web3auth.io/) under
    your Project > "Authentication" (typically in the "Custom Connections" or "Social Connections"
    tabs).
    - ![Connections on Dashboard](https://i.ibb.co/nNG9Snrq/auth-section.gif)
2.  **Use `authConnectionId`**: In your client code, when calling `connectTo` (or `connect` for
    modal flows if customizing specific dashboard connections) for an authentication flow defined on
    your dashboard (like JWT or a specific social login), you will use the `authConnectionId`
    parameter within `authParams`. This `authConnectionId` is the **Auth Connection ID** you set on
    the dashboard. (For developers migrating from v9, this `authConnectionId` replaces the
    client-side `verifier` or `verifierId` concept.)
3.  **Account Linking (`groupedAuthConnectionId`)**: If you are using aggregate connections for
    account linking (e.g., allowing a user to log in with Google or a custom JWT and access the same
    underlying Web3Auth key), you will now also specify a `groupedAuthConnectionId` in the
    `authParams` if the connection is part of such a group. This ID should match across all
    connections that you want to link to the same user account. This is also configured on the
    dashboard when setting up your connections.

## 5. Migrate React Usage to Hooks (React only)

This is a **mandatory** step for all React applications, whether you were using v9 Modal or v9
No-Modal. The class-based `Web3Auth` (or `Web3AuthNoModal`) instance should no longer be directly
managed or called within React components for connection, disconnection, or fetching user info.
Instead, use the hooks provided by `@web3auth/modal/react`.

**Key Hooks:**

- `useWeb3Auth()`: Provides access to `provider`, `user` (userInfo), `status`, `isConnected`,
  `error`, etc.
- `useWeb3AuthConnect()`: Returns `connect` (to trigger modal), `connectTo` (for direct
  connections), `status`, `error`, `provider`.
- `useWeb3AuthDisconnect()`: Returns `disconnect` function, `status`, `error`.
- `useWeb3AuthUser()`: Returns `user` (userInfo), `isConnected`.

**Setup (`Web3AuthProvider`):** As shown in Step 1, wrap your application (or relevant part) with
`Web3AuthProvider` from `@web3auth/modal/react`.

**Example Usage:**

## 6. Update Multi-Factor Authentication (MFA) Setup

The way MFA is configured has been significantly streamlined in v10.

**v9 Approach (via AuthAdapter):** In v9, MFA settings (`mfaLevel`, `mfaSettings` for granular
control) were typically configured when instantiating an `AuthAdapter` which was then passed to
`web3auth.configureAdapter()`.

**v10 Approach:** MFA configuration is now more direct:

1.  **Global `mfaSettings` (Granular Control):**

    - You can define default `mfaSettings` for all connections at the SDK initialization level.
      These settings control which MFA factors (device, backup seed phrase, social backup, password,
      passkeys, authenticator app) are enabled, their priority, and whether they are mandatory.
    - Pass the `mfaSettings` object to the `Web3Auth` constructor (for non-React) or to the
      `initParams` of `Web3AuthProvider` (for React).

    ```ts title="v10: Global mfaSettings at Initialization (Non-React)"
    import { Web3Auth, MFA_FACTOR, WEB3AUTH_NETWORK } from "@web3auth/modal";

    const web3auth = new Web3Auth({
      clientId: "YOUR_CLIENT_ID",
      web3AuthNetwork: WEB3AUTH_NETWORK.SAPPHIRE_MAINNET,
      mfaSettings: {
        [MFA_FACTOR.DEVICE]: { enable: true, priority: 1, mandatory: true },
        [MFA_FACTOR.BACKUP_SHARE]: { enable: true, priority: 2, mandatory: true },
        [MFA_FACTOR.SOCIAL_BACKUP]: { enable: true, priority: 3, mandatory: false },
        [MFA_FACTOR.PASSWORD]: { enable: true, priority: 4, mandatory: false },
        [MFA_FACTOR.PASSKEYS]: { enable: true, priority: 5, mandatory: false },
        [MFA_FACTOR.AUTHENTICATOR]: { enable: true, priority: 6, mandatory: false },
      },
    });
    await web3auth.init(); // or initModal()
    ```

    ```tsx title="v10: Global mfaSettings at Initialization (React - Web3AuthProvider)"
    // In Web3AuthProvider setup
    <Web3AuthProvider
      initParams={{
        clientId: "YOUR_CLIENT_ID",
        web3AuthNetwork: WEB3AUTH_NETWORK.SAPPHIRE_MAINNET,
        mfaSettings: {
          [MFA_FACTOR.DEVICE]: { enable: true, priority: 1, mandatory: true },
          [MFA_FACTOR.BACKUP_SHARE]: { enable: true, priority: 2, mandatory: true },
          [MFA_FACTOR.SOCIAL_BACKUP]: { enable: true, priority: 3, mandatory: false },
          [MFA_FACTOR.PASSWORD]: { enable: true, priority: 4, mandatory: false },
          [MFA_FACTOR.PASSKEYS]: { enable: true, priority: 5, mandatory: false },
          [MFA_FACTOR.AUTHENTICATOR]: { enable: true, priority: 6, mandatory: false },
        },
      }}
      chainConfig={/* ... */}
    >
      {/* ... */}
    </Web3AuthProvider>
    ```

2.  **Per-Connection `mfaLevel`:**

    - You can specify the `mfaLevel` (`"default"`, `"optional"`, `"mandatory"`, `"none"`) for a
      specific login attempt.
    - This is passed within the `authParams` (for `WALLET_CONNECTORS.AUTH`) or as a direct option
      for other specific connectors when calling `connect` (React hooks) or `connectTo`
      (class-based).

    ```ts title="v10: mfaLevel per connection (Non-React)"
    import { WALLET_CONNECTORS, AUTH_CONNECTION, MFA_LEVELS } from "@web3auth/modal";

    await web3auth.connectTo(WALLET_CONNECTORS.AUTH, {
      authParams: {
        connection: AUTH_CONNECTION.GOOGLE,
        authConnectionId: "w3a-google-demo-connection-id", // Auth Connection ID from dashboard
        mfaLevel: MFA_LEVELS.MANDATORY, // Or .OPTIONAL, .DEFAULT, .NONE
      },
    });
    ```

    ```tsx title="v10: mfaLevel per connection (React Hooks)"
    import { WALLET_CONNECTORS, AUTH_CONNECTION, MFA_LEVELS } from "@web3auth/modal";
    // Inside your component, using the connectTo function from useWeb3AuthConnect hook
    await connectTo({
      // Assuming connectTo is used for direct connection
      walletConnector: WALLET_CONNECTORS.AUTH,
      authParams: {
        connection: AUTH_CONNECTION.GOOGLE,
        authConnectionId: "w3a-google-demo-connection-id", // Auth Connection ID from dashboard
        mfaLevel: MFA_LEVELS.MANDATORY,
      },
    });
    ```

**Key `mfaSettings` properties:**

- `enable` (boolean): Enable/disable the factor.
- `priority` (number): Order of presentation (lower is earlier).
- `mandatory` (boolean): If the factor must be set up.

(Refer to the official Web3Auth MFA documentation for detailed type definitions and rules for
`mfaSettings` and `MFA_FACTOR` enum.)

This new approach removes the need for configuring MFA via a separate `AuthAdapter` instance.

## 8. Standard for React + EVM: Using Wagmi for Blockchain Interactions

For React + EVM applications using Web3Auth v10, leveraging the **Web3Auth-enhanced Wagmi provider
is the standard approach** for interacting with the blockchain. This aligns with official Web3Auth
v10 examples and ensures a smoother development experience by simplifying Wagmi setup in conjunction
with Web3Auth.

Directly using the provider obtained from Web3Auth v9 without a wrapper like Wagmi could lead to
significant refactoring or compatibility issues with the v10 provider structure. **It is strongly
advised to adopt the Web3Auth Wagmi setup and follow the patterns demonstrated in the Web3Auth v10
React examples.**

**Setup:**

1.  Install `wagmi` and its peer dependency `@tanstack/react-query`. The `viem` peer dependency is
    typically managed by Web3Auth.
    ```bash
    npm install wagmi @tanstack/react-query
    ```
2.  Wrap your application with `Web3AuthProvider` and then the Web3Auth-specific `WagmiProvider` and
    `QueryClientProvider`.

    ```tsx title="src/main.tsx or App.tsx (Web3Auth with Wagmi Provider Setup)"
    import ReactDOM from "react-dom/client";
    import { Web3AuthProvider } from "@web3auth/modal/react";
    import web3AuthContextConfig from "./web3authContext"; // Your Web3Auth context config (see Step 1 for React init)

    import { WagmiProvider } from "@web3auth/modal/react/wagmi"; // Web3Auth-specific WagmiProvider
    import { QueryClient, QueryClientProvider } from "@tanstack/react-query";

    import App from "./App";

    const queryClient = new QueryClient();

    ReactDOM.createRoot(document.getElementById("root") as HTMLElement).render(
      <Web3AuthProvider config={web3AuthContextConfig}>
        {" "}
        {/* Or initParams if not using separate context file */}
        <QueryClientProvider client={queryClient}>
          <WagmiProvider>
            {" "}
            {/* Web3Auth WagmiProvider often doesn't need explicit config prop */}
            <App />
          </WagmiProvider>
        </QueryClientProvider>
      </Web3AuthProvider>,
    );
    ```

    _Note: The `web3AuthContextConfig` in the example above refers to the configuration object you
    would define for `Web3AuthProvider`, typically containing `initParams` and `chainConfig` as
    shown in "Step 1. Update SDK Initialization" for React. If you pass `initParams` and
    `chainConfig` directly to `Web3AuthProvider` rather than through a separate config object,
    adjust accordingly._

**Usage Examples with Wagmi Hooks:**

Once Wagmi is set up, you can use its rich set of hooks within your React components for various
blockchain operations:

```tsx title="Example: Common Blockchain Interactions with Wagmi"
import { useAccount, useBalance } from "wagmi";

function MyBlockchainComponent() {
  const { address, isConnected, chainId } = useAccount();
  const { data: balance } = useBalance({ address });

  if (!isConnected) {
    return <p>Please connect your wallet using Web3Auth.</p>;
  }

  return (
    <div>
      <h3>Account Info:</h3>
      <p>Address: {address}</p>
      <p>Connected Chain ID: {chainId}</p>
      <p>Balance: {balance ? `${balance.formatted} ${balance.symbol}` : "Loading..."}</p>
    </div>
  );
}
```

This approach significantly simplifies blockchain interactions in React, making your code cleaner,
more readable, and easier to maintain, aligning with the standards set by Web3Auth v10 examples. For
more comprehensive examples covering a wider range of Wagmi hooks and blockchain interactions,
please refer to the Web3Auth Plug and Play examples repository, particularly the React examples.

## 9. Review Deprecated APIs

Ensure you are no longer using these APIs from v9:

- **`@web3auth/no-modal`**: This entire package is deprecated. Use `@web3auth/modal`.
- **`@web3auth/modal-react-hooks`**: Deprecated. React Hooks are now part of
  `@web3auth/modal/react`.
- **`@web3auth/no-modal-react-hooks`**: Deprecated. React Hooks are now part of
  `@web3auth/modal/react`.
- **`@web3auth/base`**: Functionality merged into `@web3auth/modal`.
- **`@web3auth/auth-adapter`** (and other specific adapter packages): Adapters are no longer
  configured in client code.
- **`@web3auth/account-abstraction-provider`**: Smart Account configuration is now part of
  `@web3auth/modal` options and primarily dashboard-driven.
- **`@web3auth/ethereum-provider`**, `@web3auth/solana-provider` (and other chain-specific
  providers): Not needed for Web3Auth initialization for standard chains; these are handled
  internally or via dashboard configuration.
- **Client-side `verifier`, `verifierSubIdentifier`**: Replaced by `authConnectionId` (Auth
  Connection ID from dashboard, previously analogous to verifier name/ID in v9) and
  `groupedAuthConnectionId` within `connectTo`/`connect` calls.
- **Client-side `privateKeyProvider`, `chainConfig` (for standard chains)**: Replaced by
  configuration via the Web3Auth Developer Dashboard.
- **`web3auth.configureAdapter()`**: This method is removed.
- **React class-based methods for `Web3Auth` / `Web3AuthNoModal`**: React integrations must use
  hooks from `@web3auth/modal/react`.
- **Old MFA configuration methods** (e.g., via `AuthAdapter` `loginSettings` or `adapterSettings`):
  Replaced by global `mfaSettings` in SDK init or per-connection `mfaLevel`.

## Verifying Your v10 Integration

After completing all the migration steps, it's crucial to thoroughly test your application to ensure
everything functions as expected. Here's a checklist:

- **Modal Display & UI (if using modal):** Ensure the Web3Auth Modal displays correctly with your
  configured login methods.
- **Direct Connection Flows (if using `connectTo` for custom UI):** Verify that direct connections
  to social providers or custom JWT authentications work as expected.
- **Login Methods:** Test every social login, email passwordless (if configured via OpenLogin on
  dashboard), and custom JWT connection method you have configured on the Web3Auth Dashboard.
- **Account Consistency:** If you use features like `groupedAuthConnectionId` for account linking,
  verify that logging in with different methods linked to the same identifier correctly resolves to
  the same user account and private key.
- **Blockchain Provider:** Ensure `web3auth.provider` (for class-based usage) or the provider from
  React hooks (`useWeb3AuthConnect` or `useWeb3Auth`) is available and correctly initialized for
  your target chain(s).
- **Core Blockchain Operations:**
  - Fetch user account address.
  - Fetch account balance.
  - Sign a message.
  - Send a simple transaction.
- **MFA Functionality:** Test your configured MFA flows (`mfaLevel` and/or `mfaSettings`).
- **Smart Accounts (if applicable):**
  - Verify that the Smart Account is deployed and functions as expected based on your dashboard and
    client configurations.
- **Logout Functionality:** Ensure logout clears the session and provider correctly.
- **Session Management:** Test session persistence and restoration.
- **Error Handling:** Test common error scenarios.
